# 紧凑 Skill 卡片与同步错误分流设计

## 目标

- 远程 Skill 左侧列表保持紧凑，名称和核心版本信息一眼可见。
- Supabase 健康检查使用正确的授权请求，并只在数据库结构确实缺失时提示执行 SQL。

## 远程 Skill 列表

- 卡片首行固定为：选择框、Skill 名称、远程最新版本（如 `v1`）、本地绑定版本（如 `Local v1`）。
- 首行不换行。Skill 名称占用剩余宽度，超长时省略；完整名称继续显示在右侧详情，并通过 `title` 提供悬停查看。
- 同步状态、旧版记录和来源不再占用左侧首行，统一放在右侧详情区域。
- 描述和更新时间仍保留在卡片下方，避免列表失去基本辨识信息。
- 本地 Skill 列表维持现状，本次只修正截图对应的远程列表。

## 同步健康检查与错误提示

- 远程会话检查 Storage bucket 时复用带 `apikey` 和 `Authorization` 的 Supabase 请求封装，避免健康检查自身制造缺少授权头的错误。
- 缺表、缺列或缺 bucket 属于数据库结构问题：显示“复制最新 SQL”“打开 SQL Editor”和刷新操作。
- URL、anon key、鉴权及其他网络错误属于配置或连接问题：提示检查远程同步设置，并展示技术详情；不默认引导执行 SQL。
- Skills 与会话同步使用一致的错误分流规则，避免同类错误出现不同指引。

## 测试与验收

- 客户端测试验证 bucket 健康检查请求包含 `apikey` 和 `Authorization`。
- UI 测试验证远程卡片首行不换行，只保留名称、远程版本和本地版本。
- UI 测试验证只有数据库结构类状态展示 SQL 操作，普通错误展示配置指引。
- 运行完整测试、类型检查和生产构建。
